> **Implementacija DB strane (2026-07-18):** `20260523_1000_S_rls_policies_DDL.sql` — ENABLE RLS + `tenant_isolation` policy na 22 tabele: 19 direktnih po `tenant_id` (uključujući `document.attachments` i `invoice_counters`, nastale posle pisanja plana), `appointment_services` i `supplier_contacts` preko EXISTS ka roditelju, `email_outbox` sa NULL-tenant redovima vidljivim samo platform kontekstu. Testirano kroz pravu konekciju kao `workshopadmin_app`: fail-closed bez konteksta, izolacija po tenantu, write-side odbijanje (WITH CHECK nasleđen iz USING), platform-admin vidi sve, admin bypass. Napomena za testove: `workshopadmin_admin` ne može `SET ROLE workshopadmin_app` (nije član) — testovi idu kroz posebnu konekciju kao app korisnik.

## Problem

Row-based multi-tenancy se na read strani oslanja isključivo na aplikativno filtriranje — jedan zaboravljen `WHERE tenant_id = ...` u novom backendu je direktan data leak između servisa. Kompozitni FK-ovi (2.1) štite write stranu (ne možeš upisati vezu ka tuđem redu), ali ne i čitanje.

## Predlog dizajna

**Princip:** PostgreSQL RLS kao odbrana u dubinu — policy na svim tenant-scoped domenskim tabelama, aplikacija po transakciji postavlja tenant kontekst iz JWT claima.

```sql
-- backend, na početku svake transakcije (iz claima aktivnog tenanta):
SET LOCAL app.current_tenant_id = '<uuid>';
-- za platform admin operacije:
SET LOCAL app.is_platform_admin = 'true';
```

```sql
-- obrazac policy-ja (primer za workshop.invoices):
ALTER TABLE workshop.invoices ENABLE ROW LEVEL SECURITY;

CREATE POLICY tenant_isolation ON workshop.invoices
    FOR ALL TO workshopadmin_app
    USING (
        tenant_id = NULLIF(current_setting('app.current_tenant_id', true), '')::uuid
        OR current_setting('app.is_platform_admin', true) = 'true'
    );
```

Ključne osobine ovog dizajna:

- **Migracije nisu pogođene:** RLS se primenjuje na `workshopadmin_app` (backend); `workshopadmin_admin` je vlasnik tabela i podrazumevano zaobilazi RLS (ne koristimo `FORCE`).
- **Bez konteksta = bez redova:** ako backend zaboravi `SET LOCAL`, `current_setting(..., true)` vraća NULL → policy ne propušta ništa (fail-closed).
- **Obuhvat:** domenske šeme — `customer`, `hr`, `workshop`, `warehouse` (sve tabele sa `tenant_id`) + `notification.email_outbox` (policy dopušta i `tenant_id IS NULL` redove samo platform kontekstu).
- **Van obuhvata:** `auth` i `tenant` šeme (login i izbor tenanta se dešavaju PRE postavljanja konteksta; users je globalni identitet), `codebook` (deljeni podaci), `migration`.
- **Tabele bez `tenant_id`** (`appointment_services`, `supplier_contacts`): policy preko `EXISTS` ka roditelju; `invoice_lines` i `work_order_parts` posle 2.1 imaju `tenant_id` → direktna policy.

## Trošak / rizici

- Backend mora disciplinovano postavljati `SET LOCAL` (jedno mesto u DB sloju novog backenda — connection/transaction interceptor).
- Blagi overhead po upitu (provera policy izraza); za ovaj obim zanemarljiv.
- Testovi moraju da rade sa kontekstom (ili kao admin role).
- `EXISTS` policy na child tabelama ima cenu na velikim skeniranjima — prihvatljivo, te tabele se ionako čitaju preko roditelja.

1. **RLS: DA.**
2. **Timing — DB strana odmah, backend deo uz novi backend.** Podela je bezbedna jer su dve strane nezavisne:
   - *DB strana (ide u implementacioni batch):* jedna nova skripta (`S_rls_policies_DDL.sql`, datirana posle svih tabela) sa ENABLE + CREATE POLICY za sve obuhvaćene tabele, idempotentna po postojećem obrascu. Ništa ne kvari: migracije i admin pristup rade kao i do sada (admin je vlasnik, zaobilazi RLS), stari backend se ionako ne prilagođava, aplikacija se ne koristi.
   - *Backend deo (uz novi backend):* `SET LOCAL app.current_tenant_id` interceptor u DB sloju. Fail-closed dizajn ovo pretvara u tvrdi zahtev od prvog dana — novi backend bez postavljenog konteksta ne vidi nijedan red, pa se disciplina ne može zaboraviti.
   - *Prednost redosleda DB-prvo:* policy skripta prolazi isti rebuild/test ciklus kao ostatak šeme i može se testirati odmah iz psql-a (`SET ROLE workshopadmin_app` + `SET LOCAL ...`), a novi backend nasleđuje već zaštićenu bazu umesto da se RLS naknadno kalemi.
   - *Jedina cena:* ručni upiti/debug kao `workshopadmin_app` zahtevaju postavljen kontekst (kao admin — ne zahtevaju).
