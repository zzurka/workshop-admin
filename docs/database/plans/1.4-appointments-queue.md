## Cilj

Zahtev: redosled zakazivanja se poštuje — vozila se uzimaju u rad po redosledu; dan zakazan mnogo unapred se naknadno raspoređuje; postoje dva kanala zakazivanja (radnik i portal). Trenutno appointments imaju samo `preferred_date` + `scheduled_at` — redosled unutar dana ne postoji.

## Model reda (queue)

Ključna promena mentalnog modela: **zakazuje se DAN, redosled unutar dana je eksplicitna pozicija** (ne vreme). Termin (sat) je opcion — servisi tipično rade „po redu", a preslaganje dana zakazanog unapred je promena pozicija, ne vremena.

```
scheduled_date          DATE      -- potvrđen dan servisa; NULL dok servis ne potvrdi
scheduled_time          TIME      -- opcioni precizan termin unutar dana; NULL = "po redu"
queue_position          INTEGER   -- redosled uzimanja u rad unutar (tenant, scheduled_date); CHECK (> 0)
estimated_duration_min  SMALLINT  -- procena trajanja; prefill = suma default trajanja izabranih service_types
arrived_at              TIMESTAMPTZ -- kad je vozilo fizički stiglo (walk-in dobija poziciju po dolasku)
source                  VARCHAR(10) NOT NULL DEFAULT 'staff'
                        -- CHECK IN ('staff','portal','walk_in'); presedan za VARCHAR+CHECK: email_outbox.status
```

`scheduled_at TIMESTAMPTZ` se **uklanja** — zamenjuju ga `scheduled_date` + `scheduled_time` (dva pojma koja su bila slepljena). `preferred_date` ostaje (želja mušterije sa portala, pre potvrde).

Pravila (aplikativna, dokumentovana u COMMENT):
- Potvrda zakazivanja (`confirmed`) dodeljuje `queue_position` = sledeća pozicija u danu; redosled dodele prati redosled zakazivanja (`created_at`) — time je „redosled zakazivanja se poštuje" default.
- Radnik može preslagati pozicije unutar dana (drag-and-drop pred sam dan) — zato **bez UNIQUE constrainta na poziciju**: preslaganje bi tranzitno kršilo unique (deferrable unique ne može biti partial), pa je vlasnik invarijante aplikacija. Index `(tenant_id, scheduled_date, queue_position)` služi čitanju reda.
- Portal zakazivanje: status `scheduled` + `preferred_date`; radnik potvrđuje dan → `confirmed` + `scheduled_date` + pozicija. Kapacitet dana je ljudska odluka pri potvrdi (v. odluka 4).
