## Cilj

Evidencija reklamacija na izvršen rad: prijava (u servisu ili preko portala), odluka (prihvaćena/odbijena), rešavanje — često kroz novi (garantni) radni nalog. Zakonski kontekst: potrošač ima pravo na reklamaciju (Zakon o zaštiti potrošača — rok odgovora 8 dana), pa su datumi prijave i odluke obavezni podaci.

## Izmene šeme

### A. Novi šifarnik `codebook.complaint_statuses`

Nove skripte: `20260317_0951_T_complaint_statuses_DDL.sql` + `20260317_0952_..._DML.sql`

Seed: `submitted`(1, Podneta) → `in_review`(2, U razmatranju) → `accepted`(3, Prihvaćena) / `rejected`(4, Odbijena) → `resolved`(5, Rešena) · `cancelled`(6, Povučena — kupac odustao).

### B. Nova tabela `workshop.complaints`

Nova skripta: `20260330_1340_T_complaints_DDL.sql` (posle payments 1330 iz plana 1.7)

```
id                        UUID         PK, uuidv7()
tenant_id                 UUID         NOT NULL
customer_id               UUID         NOT NULL   -- ko reklamira
work_order_id             UUID                    -- na koji rad se reklamacija odnosi
invoice_id                UUID                    -- i/ili na koju fakturu
complaint_status_id       SMALLINT     NOT NULL
description               TEXT         NOT NULL   -- šta kupac reklamira
submitted_at              TIMESTAMPTZ  NOT NULL DEFAULT NOW()  -- datum prijave (može ≠ created_at kod naknadnog unosa)
reviewed_by               UUID                    -- zaposleni koji je doneo odluku (prihvaćena/odbijena)
reviewed_at               TIMESTAMPTZ             -- datum odluke (rok 8 dana — prati aplikacija)
resolution                TEXT                    -- kako je rešena
resolution_work_order_id  UUID                    -- garantni radni nalog nastao iz reklamacije
resolved_at               TIMESTAMPTZ
notes                     TEXT                    -- interne beleške
audit kolone + is_deleted
```

Constrainti:
- `ck_workshop_complaints_target` — `CHECK (work_order_id IS NOT NULL OR invoice_id IS NOT NULL)` (reklamacija se odnosi na konkretan rad)
- Kompozitni FK-ovi (obrazac 2.1): `(tenant_id, customer_id)` → customers, `(tenant_id, work_order_id)` → work_orders, `(tenant_id, invoice_id)` → invoices, `(tenant_id, resolution_work_order_id)` → work_orders, `(tenant_id, reviewed_by)` → employees
- IX: tenant_id, customer_id, work_order_id, invoice_id, complaint_status_id, is_deleted

Portal: mušterija podnosi preko portala — `created_by` = njen `auth.users` nalog; radnik unosi za usmene prijave.

## Tok (aplikativna pravila, dokumentovati u COMMENT)

`submitted` → `in_review` → `accepted` → (garantni WO: `resolution_work_order_id`) → `resolved`; ili `rejected` (terminalno, uz obrazloženje u `resolution`); `cancelled` iz submitted/in_review. Garantni radni nalog je običan work_order (ne naplaćuje se — faktura se ne izdaje, ili ide 0 RSD linija — odluka aplikacije, ne šeme).
