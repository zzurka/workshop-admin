## A. Istorija pretplate — `tenant.tenant_subscriptions`

Nova skripta: `20260317_1040_T_tenant_subscriptions_DDL.sql` (posle user_tenants 1030 — audit FK-ovi ka users bez deferred obrasca)

```
id                    UUID        PK, uuidv7()

subscription_plan_id  UUID        NOT NULL   -- FK → subscription_plans
```

## B. Attachments — nova šema `document` + tabela

Attachmenti pripadaju svim modulima (slike vozila, dokumenti naloga, fotografije uz reklamaciju), pa dobijaju neutralnu šemu umesto kalemljenja u postojeće.

Nove skripte: `20260317_0907_S_document_DDL.sql` (šema + grantovi, obrazac postojećih S_ skripti) i `20260522_1200_T_attachments_DDL.sql`

```
id            UUID          PK, uuidv7()
tenant_id     UUID          NOT NULL   -- FK → tenants
entity_type   VARCHAR(50)   NOT NULL   -- 'vehicle', 'work_order', 'complaint', 'expense', ... (bez CHECK — lista raste)
entity_id     UUID          NOT NULL   -- polimorfna referenca, bez FK
file_name     VARCHAR(255)  NOT NULL   -- originalno ime fajla
content_type  VARCHAR(100)  NOT NULL   -- MIME tip
size_bytes        BIGINT        NOT NULL   -- CHECK (size_bytes > 0); zbir po tenantu ↔ plan.max_storage_mb (aplikativno)
storage_provider  VARCHAR(20)   NOT NULL   -- CHECK IN ('filesystem','s3'); gde fajl fizički živi (dev = filesystem, prod = S3)
storage_key       VARCHAR(500)  NOT NULL   -- ključ relativan na provider: fs putanja ili S3 object key; bucket/region/root su app config
audit kolone + is_deleted
```

- IX: tenant_id, `(entity_type, entity_id)`, is_deleted.
- Polimorfna referenca (`entity_type` + `entity_id`) namerno bez FK — jedna tabela za sve module; integritet veze čuva aplikacija, a brisanje entiteta je ionako soft.

#### Zašto zajednička polimorfna tabela, a ne alternativa (razmotreno 2026-07-17)

Razmatrane tri opcije:

1. **Kolone na tabelama** (`vehicles.photo_path`...) — najslabija: fiksan broj fajlova po redu, bez metapodataka (tip, veličina, ko je okačio), svaki novi fajl = nova kolona. Odbačeno odmah.
2. **Po-entitetska tabela** (`vehicle_attachments`, `work_order_attachments`, `complaint_attachments`...) — dobija se pravi FK (i kompozitni tenant FK iz 2.1), ali po cenu: 6+ skoro identičnih tabela već danas (vozilo, nalog, reklamacija, trošak, appointment, payslip...), svaki novi modul = nova migracija + ponovljen backend kod; kvota `max_storage_mb` po tenantu = UNION preko svih tabela; RLS policy po tabeli.
3. **Zajednička tabela bez FK** (izabrano) — jedna tabela, jedan attachment servis u modular monolitu, jedan SUM za kvotu, jedan RLS policy, novi tip entiteta = nova vrednost stringa (bez migracije).

Glavna mana opcije 3 — siročići posle brisanja entiteta i izostanak DB integriteta veze — u ovoj šemi je najvećim delom neutralisana: **sve tabele koriste soft delete**, redovi se fizički ne brišu, pa FK cascade ionako ne bi radio ništa; `tenant_id` sa pravim FK + RLS čuva tenant izolaciju (najbitniji invarijant); veza ka entitetu je aplikativna na jednom mestu (attachment servis). Uslovni/parcijalni FK po tipu u PostgreSQL-u ne postoji, a trigger-dispatch validacija po tipu bi bila krhka i održavana ručno — svesno izostavljena. Isti polimorfni obrazac već koristi `email_outbox.related_entity_*` (deo C ovog plana).

Opcija 2 postaje bolja tek ako različiti tipovi priloga dobiju bitno različita pravila/kolone (npr. verzionisani potpisani ugovori) — tada se taj tip izdvoji u svoju tabelu, a ostatak ostaje u zajedničkoj.

#### Dualni storage (filesystem za dev, S3 za prod)

`storage_provider` je **po redu**, ne globalni config: fajlovi upisani na filesystem ostaju čitljivi i posle prelaska na S3 (backendi koegzistiraju), a migracija fs → S3 može ići postepeno (kopiraj fajl → update provider+key po redu). Aplikativni storage sloj bira implementaciju po vrednosti kolone; bucket/region odnosno root direktorijum su konfiguracija aplikacije, ne baza. Konvencija ključa (aplikativna, ista za oba providera): `{tenant_id}/{entity_type}/{attachment_id}` — tenant prefiks olakšava kvote i eventualni export tenanta.

## C. `notification.email_outbox` — poreklo poruke (izmena u mestu: `20260521_1010`)

```
template_code        VARCHAR(50)   -- koji template je proizveo poruku; bez FK (informativno)
related_entity_type  VARCHAR(50)   -- npr. 'invoice', 'appointment' — isti polimorfni obrazac kao attachments

```

+ IX `(related_entity_type, related_entity_id)`. Omogućava dijagnostiku („koji mejlovi su otišli za ovu fakturu") i aplikativnu idempotentnost (provera pre enqueue).
