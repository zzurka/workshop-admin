## Izmene

### A. `auth.users` (izmena u mestu: `20260317_1010`)

1. **Email unique postaje partial** — briše se `uq_auth_users_email UNIQUE (email)`, dodaje se:
   ```sql
   CREATE UNIQUE INDEX uq_auth_users_email ON auth.users (LOWER(email)) WHERE NOT is_deleted;
   ```
   Soft-obrisan user oslobađa email (konzistentno sa roles); `LOWER()` štiti od duplikata po velikim/malim slovima (aplikacija i dalje normalizuje na lowercase pri upisu). Postojeći `ix_auth_users_email` se briše (partial unique ga pokriva).
2. **`email` VARCHAR(255) → VARCHAR(320)** — ujednačeno sa `external_logins`/`email_outbox` (RFC max je 320).
3. **Verifikacija emaila:** `email_verified_at TIMESTAMPTZ` (NULL = neverifikovan). Obavezno za samoregistraciju mušterija; postojeći nalozi koje kreira radnik dobijaju verifikaciju kroz prvi login link (aplikativno).
4. **Lockout:** `failed_login_count SMALLINT NOT NULL DEFAULT 0` + `locked_until TIMESTAMPTZ`. Aplikacija: increment na neuspeh, reset na uspeh, zaključavanje posle N pokušaja (N i trajanje su config, ne DB).
5. **MFA checkovi:** `ck_auth_users_mfa_method CHECK (mfa_method IS NULL OR mfa_method IN ('totp','sms','email'))` + `ck_auth_users_mfa_enabled CHECK (NOT mfa_enabled OR mfa_method IS NOT NULL)`.

### E. RLS za `roles`, `role_permissions`, `user_roles` (dopuna 2026-07-18)

Nova skripta: `20260523_1010_S_rls_policies_auth_DDL.sql` (posle glavne RLS skripte 2.2)

`auth.roles` nije čisto globalni katalog: `tenant_id IS NULL` = sistemske role, postavljen = **custom role tenanta** (kreira ih tenant admin kroz redovan API, sa postavljenim kontekstom). Zaboravljen `WHERE tenant_id` tu znači da tenant B vidi/menja custom role tenanta A — isti threat model kao domenske tabele u [2.2](2.2-rls.md). Policy dizajn se razlikuje od domenskog obrasca jer tabele mešaju deljene i tenant redove — **razdvojene policy po komandi** (jedan `FOR ALL` bi kroz `USING` dozvolio tenant kontekstu i izmenu globalnih rola):

```sql
-- auth.roles — čitanje: globalne role su deljeni katalog (vidljive bez konteksta,
-- kao codebook); custom role samo u svom tenant kontekstu
CREATE POLICY roles_select ON auth.roles FOR SELECT TO workshopadmin_app
    USING (tenant_id IS NULL
           OR tenant_id = NULLIF(current_setting('app.current_tenant_id', true), '')::uuid
           OR current_setting('app.is_platform_admin', true) = 'true');

-- auth.roles — pisanje: tenant kontekst sme samo svoje redove; globalne samo platforma
-- (INSERT: WITH CHECK; UPDATE: USING + WITH CHECK; DELETE: USING — isti izraz)
tenant_id = NULLIF(current_setting('app.current_tenant_id', true), '')::uuid
OR current_setting('app.is_platform_admin', true) = 'true'
```

- `auth.role_permissions` (nema `tenant_id`): SELECT preko `EXISTS` ka roli sa uslovom čitanja; write preko `EXISTS` ka roli sa uslovom pisanja — dodela permisija custom roli je moguća samo u njenom tenant kontekstu, a sistemskim rolama samo platformi.
- `auth.user_roles` (`tenant_id IS NULL` = platform dodela): obrazac kao `email_outbox` — tenant kontekst vidi/piše samo redove svog tenanta; NULL redovi (platform dodele) vidljivi/pisivi samo platform kontekstu. Bez bezuslovne vidljivosti.

**Sprega sa auth tokom (obaveza za novi backend, zabeleženo i u backend planu):** građenje claim-ova pri izdavanju tokena čita ove tabele pre nego što JWT postoji. Transakcija izdavanja tokena u auth modulu se zato izvršava sa `SET LOCAL app.is_platform_admin = 'true'` — jedno privilegovano, revidirano mesto (izbegava i chicken-and-egg: platform status korisnika se utvrđuje čitanjem NULL-tenant redova `user_roles`). Ostatak auth šeme (users, tokeni, MFA, login_history) ostaje van RLS-a — nema tenant dimenziju.

**Test dopuna:** custom rola tenanta A nevidljiva/neizmenjiva u kontekstu tenanta B; globalna rola čitljiva bez konteksta ali UPDATE iz tenant konteksta odbijen; `role_permissions` prati rolu; `user_roles` platform dodela nevidljiva tenant kontekstu.
